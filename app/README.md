# Open Campus — App (Flutter)

The Flutter client (Android / iOS / web). v1 = a polished vertical slice: login →
dashboard showing your **Results** (CGPA trend + table) and **Attendance**
(per-course), with the design system, on-device last-view cache, and a soft
re-login prompt on session expiry.

> The `lib/` source and `pubspec.yaml` are committed; the platform folders
> (`web/`, `android/`, …) are generated by `flutter create` (see setup).

## Prerequisites
- **Git for Windows** (https://git-scm.com/downloads/win).
- **VS Code** + the **Flutter** extension (by Dart-Code).
- **Chrome** (for the web build — fastest to run).

## Install Flutter (via VS Code — current official method)
The old "download the zip" method is deprecated. Install through VS Code:
1. Extensions (`Ctrl+Shift+X`) → search **Flutter** → Install (pulls in Dart too).
2. Command Palette (`Ctrl+Shift+P`) → **Flutter: New Project** → **Download SDK**.
3. Pick an SDK folder **without spaces** (e.g. `C:\src` → creates `C:\src\flutter`).
   ⚠️ NOT a path with spaces.
4. **Clone Flutter** → wait → **Add SDK to PATH** → OK.
5. Restart all terminals and VS Code, then verify: `flutter doctor -v`.

> The "New Project" flow only triggers the SDK download. If it forces you to
> create a project, make a throwaway one in e.g. `C:\src\tmp` and delete it —
> we already have THIS app; don't create a new one over it.

## First-time setup for THIS app

```bash
cd "d:/Web/Open Campus/app"

# 1) Generate the platform scaffolding AROUND the existing lib/ + pubspec.
#    This does NOT overwrite lib/ or pubspec.yaml.
flutter create .

# 2) Get dependencies
flutter pub get
```

## Run (web)

The backend must be running first (see ../backend/README.md). Then:

```bash
# Pin the web port to 5000 so it matches the backend's CORS allow-list.
flutter run -d chrome --web-port=5000
```

The app opens in Chrome. Log in with your UCAM ID + password → dashboard.

> **Backend URL:** defaults to `http://127.0.0.1:8000` on web. Override with
> `--dart-define=OC_API_BASE=https://your-host` when you deploy the backend.

## Architecture (lib/)

```
core/
  api/        api_config (base url per platform), api_client (dio + ApiResult)
  auth/       token_storage (secure)
  cache/      local_cache (on-device last-view — the stateless-server model)
  theme/      app_theme (design system: colors, type, spacing, light/dark)
  providers   Riverpod singletons (api client, token, cache)
features/
  auth/       auth_controller (login/restore/logout), login_screen
  dashboard/  models, dashboard_controller (live → cache fallback), dashboard_screen
shared/       widgets (SectionCard, FreshnessChip, Skeleton, ReloginBanner)
main.dart     routes login ⇄ dashboard on auth state
```

## How data freshness works
- Each section fetches **live** from the backend. On success it shows a green
  "Live" chip and caches the payload on-device.
- If the live fetch fails, it falls back to the **on-device cache** with a
  "Synced Xm ago" chip. If the UCAM session expired, a soft re-login banner
  appears — the cached view stays visible (no hard kick).
- Logout clears the on-device cache.
